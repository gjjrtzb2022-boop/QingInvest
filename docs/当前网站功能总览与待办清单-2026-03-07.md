## 三、待解决问题（按优先级）

## P0（上线前必须解决）

1. **部署形态与实时能力冲突**
- 现有 CI 发布到 GitHub Pages，`npm run build` 走静态导出（`NEXT_OUTPUT_EXPORT=1`）。
- 但 stocks 依赖 `/api/stocks/realtime` 与 `/api/stocks/kline` 的服务端接口。
- 结论：静态托管下实时行情/K线接口不可用或不可持续，应迁移到支持 Node Runtime 的部署（如 Vercel/自建）。

2. **认证链路稳定性未完成验收**
- OTP 功能受环境变量和 Supabase 邮件/短信配置影响。
- 当前代码保留“暂缓/待开通”分支，属于“可开发态”，不是稳定生产态。

---

## P1（近期应完成）

1. **文章查询仍以文件系统为主，不是数据库查询主链路**
- 目前 `/articles` 仍由 `lib/articles.ts` 读取 `content/articles/*.md`。
- 对数千/上万篇文章时，筛选、搜索、构建与冷启动压力会明显增大。
- 建议：切换到 PostgreSQL 查询层（分页、索引、FTS）。

2. **选股业务有“实时行情 + 静态策略”混合问题**
- 行情实时，但策略评分、候选池逻辑、部分指标仍是规则启发式。
- `topSummary` 等统计存在 hash/固定逻辑，不是严格行情推导。

3. **股票池覆盖有限**
- 当前股票池来自 `lib/stocks.ts` 的固定列表（约 33 只），非全市场。
- 若目标是公开选股器，需要可维护的全市场池和定时更新。

4. **1秒轮询成本与风控风险**
- 高频轮询会增加第三方接口压力与限流概率。
- 需加入分级刷新策略（前台1s、后台降频、tab不可见降频、失败退避）。

---

## P2（中期优化）

1. **K线长期数据在不同源下仍有上限差异**
- 虽已把请求上限放大，但不同数据源实际可返回长度不完全一致。
- 部分股票在月/季/年仍受上游接口可用历史长度限制。

2. **全文搜索能力未落地**
- 目前搜索主要基于列表聚合字段匹配。
- 尚未启用数据库 FTS / 语义检索。

3. **统一可观测性不足**
- 已有前端同步健康统计，但缺少服务端统一指标看板（接口延迟、失败分布、重试耗时、来源健康度历史曲线）。

4. **支付/订阅/权限分层尚未开始**
- 若目标是收费产品，还需补充：套餐、权限、风控、账单、审计日志。

---

## 四、建议的下一阶段顺序（执行版）

1. **先定部署架构**：确定从 GitHub Pages 切到支持 API 的运行环境（这是实时能力前提）。
2. **修认证闭环**：先把邮箱 OTP 做成稳定可验收链路（发送、验证、回调、异常提示）。
3. **文章库切 DB 查询**：把 `/articles` 查询主链路从文件读取切到 Postgres。
4. **选股器去“伪指标”**：把 hash/模板化统计替换为真实可追溯计算。
5. **再扩模块**：分析/预警/AI/财务在统一数据层上并行推进。

---

## 五、结论

你现在的网站已经不是“静态展示页”，而是进入了“**可运营雏形**”阶段：
- 强项：文章库、阅读交互、同步框架、选股器实时行情/K线。
- 短板：生产部署形态、认证稳定性、数据库查询主链路、商业化能力（订阅/权限/审计）。

只要先把 P0/P1 收口，这个项目就能从“开发演示态”进入“可正式上线迭代态”。
